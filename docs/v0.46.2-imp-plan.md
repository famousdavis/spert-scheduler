# SPERT Scheduler v0.46.2 — Implementation Plan (v2): Extend useBufferedField to remaining controlled name inputs

## Changes from v1

Claude Chat reviewed v1 and asked for four adjustments, all folded in here:

1. **Enter handler now uses `inputRef.current?.blur()` defensively before `setIsEditing(false)`** — removes the dependency on React's synthetic-blur-on-unmount behavior, which is version-inconsistent. This was an "open question" in v1; in v2 it is no longer optional.
2. **`useBufferedField` gains a `reset()` method** in its return value — replaces the v1 `setLocalValueRef` ref-bridge pattern. The hook contract is now explicit ("callers may call `reset()` from inside an `onCommit` callback to resync the buffer to the current external value") and there is no circular-dep dance at the InlineEdit call site.
3. **`MilestoneNameInput` is an inline non-exported component inside `MilestonePanel.tsx`** — not a new file.
4. **Scenario rename ordering verification is a hard manual-checklist item**, not an assumption. The v1 plan reasoned about the ordering and concluded it was preserved (`onSave` runs before `setIsEditing(false)` in both old and new `commit`/`handleCommit`); v2 still demands that you confirm empirically before shipping.

v2 also marks the post-PR test count as tentative (1572/73 estimated, confirm post-run).

## Background

v0.46.1 introduced `src/ui/hooks/use-buffered-field.ts` and applied it to:

- `UnifiedActivityRow` activity name input
- `BandHeaderRow` section header name input

Both inputs previously suffered from a structural cloud-sync echo race: per-keystroke controlled input + per-keystroke debounced cloud save → Firestore server-ack snapshot (`hasPendingWrites: false`) → `mergeProject` → re-render → keystrokes typed since debounce-fire are silently overwritten. The hook buffers the value in local state while focused (focus guard via ref), commits change-aware against a focus-time snapshot (so no-edit blurs don't overwrite remote updates), and provides `revertValue()` for Escape with a `suppressNextBlur` flag.

v0.46.2 extends the hook to the remaining controlled string inputs in the codebase that carry the same structural exposure.

### Scope decision: drop the modal from v0.46.2

The ActivityEditModal's name and notes fields use `useState(prop)` initializers without a corresponding sync `useEffect`. While the modal is open, external store changes (cloud sync, undo, collaborator edits) do not flow into the modal's local state. The modal commits all fields atomically on the Save button via `handleSave` ([ActivityEditModal.tsx:404-449](src/ui/components/ActivityEditModal.tsx)). There is no per-keystroke commit, and there is no re-render path that can clobber typed characters.

The modal does carry a different concern: when the user clicks Save, the modal overwrites whatever is in the store, regardless of whether a remote update arrived in the meantime. This is "last-Save-wins," which is the documented v0.46.1 semantic generalized to a Save button — not a bug, and not something `useBufferedField` would change.

The other side of the same coin: if the user has uncommitted text in the (now-buffered) grid name input and opens the modal, the modal initializes from `activity?.name` (the committed store value), not from the grid's typed-but-uncommitted value. By the time the modal renders, the click-away has triggered the grid input's blur and the store has been updated. Confirm this empirically during manual verification but assume correct.

If a user reports the modal mishandling input in cloud mode, that becomes a v0.46.3 issue.

The `NewProjectDialog`, `NewScenarioDialog`, and `CloneScenarioDialog` follow the same Save-button pattern and are similarly unaffected.

## Hook change: add `reset()` to `useBufferedField`

`useBufferedField` gains one new method in its return:

```ts
export interface BufferedField {
  localValue: string;
  setLocalValue: (v: string) => void;
  handleFocus: () => void;
  handleBlur: () => void;
  revertValue: () => void;
  /**
   * Force-resync the buffer to the current externalValue, overriding the
   * focus guard. Intended for callers that need to reset the buffer after
   * a rejected commit (e.g., InlineEdit rejecting whitespace-only input):
   * call from inside an `onCommit` callback to discard the typed value
   * and snap back to the canonical external value.
   *
   * Distinct from `revertValue()`:
   *   - `revertValue()` resets to the FOCUS-TIME snapshot and sets the
   *     suppressNextBlur flag (for Escape — the caller is about to .blur()
   *     synchronously and wants to skip the impending commit).
   *   - `reset()` resets to the CURRENT externalValue and does NOT touch
   *     the suppress flag (for post-commit rejection — the commit has
   *     already run and decided not to save; we just want the buffer
   *     re-synced for the next focus cycle).
   */
  reset: () => void;
}

// ...

const reset = useCallback(() => {
  setLocalValue(externalValue);
}, [externalValue]);

return { localValue, setLocalValue, handleFocus, handleBlur, revertValue, reset };
```

The `useCallback` dep is `externalValue` — `reset`'s identity changes whenever the external value changes. This is fine for JSX handlers and fine when called from inside an `onCommit` closure: the closure captures the `reset` from the same render, which captures the same `externalValue` that handleCommit captured.

**New hook test (8th in `use-buffered-field.test.ts`)** — locks the contract:

> **`reset()` resyncs the buffer to externalValue and bypasses the focus guard.**
>
> Render with `externalValue = "A"`. Call `handleFocus`. Call `setLocalValue("typed")`. Assert `localValue === "typed"`. Call `reset()`. Assert `localValue === "A"`. Call `handleBlur`. Assert `onCommit` not called (localValue equals focusedSnapshot, change-aware guard holds).

A second test for the externalValue-changed-then-reset case is worth adding:

> **`reset()` uses the current externalValue, not the focus-time snapshot.**
>
> Render with `externalValue = "A"`. Call `handleFocus` (snapshot = "A"). Call `setLocalValue("typed")`. Rerender with `externalValue = "B"` (focus guard suppresses, localValue stays "typed"). Call `reset()`. Assert `localValue === "B"` (current external, not snapshot "A"). Call `handleBlur`. Assert `onCommit("B")` called once (localValue "B" !== focusedSnapshot "A" → commit fires).

## Implementation

### 1. Migrate `MilestonePanel.tsx`

Hooks can't be called inside `.map()`. Extract an inline non-exported `MilestoneNameInput` component at the top of `MilestonePanel.tsx` (above the exported `MilestonePanel`), keyed on `m.id` at the call site so each row gets its own hook instance.

```tsx
// In MilestonePanel.tsx — non-exported, lives in the same file.
interface MilestoneNameInputProps {
  milestoneId: string;
  name: string;
  disabled: boolean;
  onCommit: (milestoneId: string, name: string) => void;
}

function MilestoneNameInput({ milestoneId, name, disabled, onCommit }: MilestoneNameInputProps) {
  const inputRef = useRef<HTMLInputElement>(null);
  const handleCommit = useCallback(
    (next: string) => onCommit(milestoneId, next),
    [milestoneId, onCommit]
  );
  const {
    localValue, setLocalValue, handleFocus, handleBlur, revertValue,
  } = useBufferedField(name, handleCommit);

  return (
    <input
      ref={inputRef}
      id={`milestone-name-${milestoneId}`}
      type="text"
      name="milestoneName"
      autoComplete="off"
      aria-label="Milestone name"
      value={localValue}
      onChange={(e) => setLocalValue(e.target.value)}
      onFocus={handleFocus}
      onBlur={handleBlur}
      onKeyDown={(e) => {
        if (e.key === "Escape") {
          e.preventDefault();
          revertValue();
          inputRef.current?.blur();
        }
        // No Enter handling — milestone names commit on blur/Tab like
        // any normal input. There is no "next field" tab-order contract
        // to enforce here.
      }}
      disabled={disabled}
      className="flex-1 min-w-0 px-2 py-1 text-sm font-medium border border-transparent hover:border-gray-200 dark:hover:border-gray-600 dark:bg-gray-800 dark:text-gray-100 rounded focus:border-blue-400 focus:outline-none disabled:opacity-60"
      placeholder="Milestone name"
    />
  );
}
```

In the existing `MilestonePanel` render — wrap the commit closure once with `useCallback` to keep `MilestoneNameInput`'s commit dep stable across parent re-renders:

```tsx
const handleMilestoneNameCommit = useCallback(
  (id: string, name: string) => onUpdateMilestone(id, { name }),
  [onUpdateMilestone]
);

// ... inside the .map((m) => ...) body, replace the existing <input> with:
<MilestoneNameInput
  key={m.id}                              // required for hook identity on reorder/delete
  milestoneId={m.id}
  name={m.name}
  disabled={!!isLocked}
  onCommit={handleMilestoneNameCommit}
/>
```

### 2. Migrate `InlineEdit.tsx`

```tsx
export function InlineEdit({
  value, onSave, className = "", placeholder = "Untitled",
  inputClassName = "", name = "inlineEdit", ariaLabel,
}: InlineEditProps) {
  const [isEditing, setIsEditing] = useState(false);
  const inputRef = useRef<HTMLInputElement>(null);

  const handleCommit = useCallback(
    (next: string) => {
      const trimmed = next.trim();
      if (trimmed && trimmed !== value) {
        onSave(trimmed);
      } else {
        // Empty-after-trim or unchanged-after-trim → no save. Reset the
        // buffer back to `value` so a re-open of edit mode shows the
        // canonical value, not the user's whitespace-only typed input.
        reset();
      }
      setIsEditing(false);
    },
    // `reset` is declared by useBufferedField below — TypeScript will
    // catch the use-before-declare if you swap the order. See note.
    [value, onSave /* reset added by hook destructure below */]
  );

  const {
    localValue, setLocalValue, handleFocus, handleBlur, revertValue, reset,
  } = useBufferedField(value, handleCommit);

  // Auto-focus + select when entering edit mode.
  useEffect(() => {
    if (isEditing && inputRef.current) {
      inputRef.current.focus();
      inputRef.current.select();
    }
  }, [isEditing]);

  if (isEditing) {
    return (
      <input
        ref={inputRef}
        type="text"
        name={name}
        autoComplete="off"
        aria-label={ariaLabel ?? placeholder}
        value={localValue}
        onChange={(e) => setLocalValue(e.target.value)}
        onFocus={handleFocus}
        onBlur={handleBlur}
        onKeyDown={(e) => {
          if (e.key === "Enter") {
            e.preventDefault();
            // Defensive: synchronously fire blur on the focused input.
            // handleBlur runs → handleCommit runs (onSave + setIsEditing(false)).
            // Avoids relying on React's synthetic-blur-on-unmount behavior.
            inputRef.current?.blur();
          } else if (e.key === "Escape") {
            e.preventDefault();
            // revertValue() sets suppressNextBlur + queues a state revert
            // to the focus-time snapshot. The .blur() that follows fires
            // handleBlur, which sees the suppress flag and returns early
            // (no commit). setIsEditing(false) is called explicitly here
            // because handleCommit is NOT invoked on the suppressed blur
            // path.
            revertValue();
            inputRef.current?.blur();
            setIsEditing(false);
          }
        }}
        className={`border border-blue-400 rounded px-2 py-1 focus:outline-none focus:ring-1 focus:ring-blue-400 ${inputClassName}`}
        placeholder={placeholder}
      />
    );
  }

  return (
    <span
      onClick={() => setIsEditing(true)}
      className={`cursor-pointer hover:bg-gray-100 rounded px-1 py-0.5 transition-colors ${className}`}
      title="Click to edit"
    >
      {value || placeholder}
    </span>
  );
}
```

**Notes on this code:**

1. **`handleCommit` dep array.** `reset` from the hook destructure is used inside `handleCommit`. TypeScript sees `reset` used before declared — it's hoisted but the linter (`react-hooks/exhaustive-deps`) will flag the missing dep. Solution: declare an empty placeholder in the dep list and let the lint warning guide you, OR restructure as below (recommended):

   Restructure: move the `useBufferedField` call BEFORE `handleCommit`, and use a ref to bridge `onCommit` into the hook. This is the same shape `useCallback` users typically resort to when a callback needs access to another callback returned from a hook called on the same line. Concrete pattern:

   ```ts
   // Forward declaration via ref. `handleCommit` reads from the ref, which
   // is set after `useBufferedField` returns.
   const commitRef = useRef<((next: string) => void) | null>(null);
   const stableOnCommit = useCallback((next: string) => {
     commitRef.current?.(next);
   }, []);

   const {
     localValue, setLocalValue, handleFocus, handleBlur, revertValue, reset,
   } = useBufferedField(value, stableOnCommit);

   // Now we can reference `reset` cleanly.
   const handleCommit = useCallback(
     (next: string) => {
       const trimmed = next.trim();
       if (trimmed && trimmed !== value) {
         onSave(trimmed);
       } else {
         reset();
       }
       setIsEditing(false);
     },
     [value, onSave, reset]
   );

   useEffect(() => {
     commitRef.current = handleCommit;
   }, [handleCommit]);
   ```

   This is structurally similar to the v1 `setLocalValueRef` pattern but moves the indirection to a more honest place: the hook's `onCommit` parameter is the unstable thing (it depends on `reset`), so the bridge is on `onCommit`, not on `setLocalValue`. The hook's contract stays clean (it just exposes `reset`), and the InlineEdit call site explicitly handles the dep cycle.

   **Decision:** ship the ref-bridge restructure. It's the cleanest way to satisfy the dep array and the linter.

2. **Defensive `.blur()` on Enter.** No longer optional. Always present. Removes any dependency on React firing synthetic blur during conditional unmount of a focused element — version-dependent behavior that has bitten people before.

3. **Escape calls `setIsEditing(false)` explicitly.** Required because `revertValue()` sets `suppressNextBlur`, so the `.blur()` that follows does NOT invoke `handleCommit`. Without the explicit `setIsEditing(false)`, the input would stay in edit mode after Escape.

4. **The two-step ordering on Escape (revertValue → blur → setIsEditing).** `revertValue()` queues a `setLocalValue(focusedSnapshot.current)`. The `.blur()` fires synchronously and `handleBlur` reads `suppressNextBlur === true` and returns. `setIsEditing(false)` then schedules re-render. The conditional unmounts the input. The `useBufferedField` `useEffect` on the next render sees `isFocused.current === false` (set by `handleBlur`) and re-syncs `localValue` to `externalValue` (which equals `value`). Input visually reverts cleanly. The intermediate state (`localValue` briefly equal to `focusedSnapshot`) is invisible.

5. **The commit-then-setIsEditing(false) ordering is preserved from the original `commit()`.** Both old `commit()` and new `handleCommit` call `onSave(trimmed)` first, `setIsEditing(false)` second. This matters because `ScenarioTabs` passes `onSave={(name) => { onRename(scenario.id, name); onStartEditing(""); }}`, and the original behavior was for `onStartEditing("")` (parent state) to run before `setIsEditing(false)` (InlineEdit local state). v2 preserves this. **Verify on the manual checklist anyway** — see [§ Manual verification](#manual-verification).

## CHANGELOG entry (v0.46.2)

```markdown
## 0.46.2 — <date>

### Fix — milestone names + project/scenario rename inputs no longer drop characters in cloud mode

v0.46.1 fixed the activity grid name input and section header name input against the cloud-sync echo race. v0.46.2 extends the same fix to the three remaining controlled name inputs that exhibited the same structural exposure:

- Milestone name (`MilestonePanel`) — was a fully controlled per-keystroke input, identical to the pre-v0.46.1 activity name pattern.
- Project rename and scenario tab rename — both backed by the shared `InlineEdit` component, which had the same `useState` + sync `useEffect` pattern as pre-v0.46.1 BandHeaderRow (sync without focus guard, vulnerable to mid-typing overwrite from server-ack snapshots).

`InlineEdit` was migrated to use `useBufferedField` internally; both call sites inherit the fix. The reject-empty-trim policy and edit-mode toggle remain `InlineEdit`'s responsibility. `useBufferedField` gained a new `reset()` method in its return value to support the rejection path — distinct from `revertValue()` (which is the Escape-time revert to the focus-time snapshot with `suppressNextBlur` armed).

Internal hardening: `InlineEdit`'s Enter handler now calls `inputRef.current?.blur()` explicitly before exiting edit mode. The prior implementation relied on React firing a synthetic blur event during the conditional unmount of the focused input — version-inconsistent behavior. The explicit `.blur()` removes the dependency.

Same behavioral semantics as v0.46.1 apply to these sites: commit timing is now per-edit-session (focus loss / Enter / click-away) instead of per-keystroke; Escape reverts to focus-time value; external rename while focused is suppressed (last-blur-wins); a focused-and-blurred-without-typing field does not overwrite remote updates that arrived during focus.

### Not migrated (intentionally)

The ActivityEditModal name field and notes textarea were considered for migration but are not migrated. The modal uses `useState(prop)` initializers without a sync `useEffect`; while open, the modal's local state is not subscribed to external store updates. It commits all fields atomically on the Save button. There is no per-keystroke commit and no re-render path that can drop characters. Migrating it to `useBufferedField` would not fix a real bug; the hook's focus-guard semantics don't add anything when the commit trigger is a button.

Other one-shot dialog inputs (`NewProjectDialog`, `NewScenarioDialog`, `CloneScenarioDialog`) follow the same Save-button pattern and are similarly unaffected.
```
